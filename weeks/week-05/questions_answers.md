**1. В чем главное концептуальное отличие GraphQL от REST?**
Главное отличие в том, что REST — это архитектура, где у каждой сущности или ресурса есть свой эндпоинт (например, `/users`, `/posts`), и сервер решает, какие данные отдавать. GraphQL же имеет единую точку входа (`/graphql`), и клиент сам в запросе указывает, какие именно поля ему нужны. Это решает проблему избыточности или недостаточности данных (overfetching/underfetching).

**2. Что такое Schema Definition Language (SDL)?**
Это синтаксис, на котором разработчик описывает структуру и типы данных, доступные в API. Например, так:
```graphql
type User {
  id: ID!
  name: String!
  posts: [Post!]!
}
```
SDL служит контрактом между сервером и клиентом.

**3. Чем Query отличается от Mutation? Могут ли они выполняться параллельно?**
*   **Query** предназначен для чтения (получения) данных. Он выполняется параллельно, потому что не должен менять состояние сервера (идемпотентен).
*   **Mutation** предназначен для изменения данных (создание, обновление, удаление).
Выполняются они **последовательно (строго друг за другом)**, чтобы избежать состояния гонки (race conditions), когда два запроса на изменение пытаются одновременно поменять одни и те же данные.

**4. Что такое "резолвер" (resolver) и кто его вызывает?**
Резолвер — это функция-обработчик, которая объясняет GraphQL, как найти данные для конкретного поля в схеме. У каждого поля схемы есть резолвер.
Вызывает его **GraphQL-движок (бэкенд)** в момент выполнения запроса. Он проходит по дереву запроса и запускает соответствующие резолверы, чтобы собрать финальный результат.

**5. Как решается проблема N+1 запросов в GraphQL (dataloader)?**
Проблема N+1 возникает, когда мы сначала запрашиваем список пользователей (1 запрос), а потом для каждого пользователя резолвер отдельно запрашивает его заказы (N запросов).
Решается это с помощью пакета **DataLoader**. Он группирует (batches) все запросы по ключам (например, IDs пользователей) и объединяет их в один запрос к базе данных или API, а также кэширует результаты в рамках одного HTTP-запроса.

**6. Почему в GraphQL сложно реализовать кэширование на уровне HTTP (как в REST)?**
В REST кэширование простое, потому что ресурсы имеют уникальные URL. Мы можем закэшировать ответ по адресу `/posts/1`, так как знаем, что там всегда пост с id=1.
В GraphQL все запросы идут на один и тот же URL (`/graphql`), отличаются только телами запросов (POST-данными). HTTP-кэширование работает на уровне URL, а не содержимого запроса, поэтому закэшировать ответ просто так не получится. Здесь используется более сложное кэширование на клиенте (например, нормализованный кэш в Apollo Client) или персистентные запросы.
