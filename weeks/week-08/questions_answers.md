## Вопросы для самопроверки

**1. Что такое Server Streaming? Приведите пример использования.**

Server Streaming — это когда клиент отправляет один запрос, а сервер отвечает потоком сообщений. Соединение остается открытым, и сервер отправляет данные частями по мере готовности.

Пример: подписка на обновления курса валют, лента новостей, передача большого файла, логи сервера.

---

**2. Чем Server Streaming отличается от просто получения большого списка (List) в одном ответе?**

При обычном List сервер собирает все данные в один ответ — клиент ждет, пока всё загрузится, и получает всё сразу.

При Streaming данные приходят частями по мере готовности. Клиент может начать обрабатывать первые данные, не дожидаясь остальных. Это экономит память и снижает задержку до первого байта.

---

**3. Какие метрики производительности существуют (Latency, Throughput, CPU Usage)?**

- **Latency** — время от отправки запроса до получения ответа. Для стриминга часто меряют "time to first byte"
- **Throughput** — количество запросов в секунду (RPS), которые сервер может обработать
- **CPU Usage** — нагрузка на процессор при обработке запросов. gRPC тратит меньше CPU на парсинг JSON

---

**4. Почему важно переиспользовать канал (Channel) в gRPC клиенте при замерах производительности?**

Создание нового канала на каждый запрос — это дорого. Канал устанавливает TCP соединение, выполняет handshake, настраивает HTTP/2. При замерах производительности мы хотим измерить время работы самого сервиса, а не накладные расходы на соединение. Переиспользование канала дает честные результаты.

---

**5. В каких сценариях REST может оказаться быстрее или удобнее gRPC?**

REST удобнее когда:
- Нужно быстро протестировать через браузер или curl
- Работаем с простыми данными, где Protobuf не дает большого выигрыша
- API используется внешними клиентами (JavaScript в браузере плохо работает с gRPC)
- Нужна простота отладки и кэширование на уровне HTTP

REST может быть быстрее для очень простых запросов, где накладные расходы gRPC (бинарная сериализация) перевешивают выгоду.

---

**6. Что такое "Head-of-Line Blocking" и как HTTP/2 (и gRPC) решает эту проблему?**

Head-of-Line Blocking — проблема HTTP/1.1, когда один медленный запрос блокирует все остальные, потому что соединение обрабатывает запросы последовательно.

HTTP/2 решает это через мультиплексирование — несколько запросов идут по одному TCP соединению параллельно, разбиваясь на независимые фреймы. Если один запрос тормозит, остальные продолжают передаваться. gRPC использует HTTP/2, поэтому наследует это преимущество.
