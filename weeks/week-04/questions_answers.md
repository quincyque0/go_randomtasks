# Ответы на вопросы по Сагам и распределенным транзакциям

## 1. Почему в микросервисах нельзя просто использовать `BEGIN TRANSACTION ... COMMIT`?

В микросервисной архитектуре каждый сервис имеет свою собственную базу данных. ACID-транзакции работают только в пределах одной БД. Не существует распределенного менеджера транзакций, который мог бы координировать коммит или откат сразу в нескольких независимых базах данных разных сервисов. Использование 2PC (Two-Phase Commit) в распределенной системе приводит к:
- Блокировкам ресурсов на длительное время
- Снижению производительности
- Проблемам с масштабируемостью

## 2. Разница между Оркестрацией и Хореографией

| Характеристика | Оркестрация | Хореография |
|----------------|-------------|-------------|
| **Суть** | Центральный координатор управляет шагами | Сервисы общаются через события |
| **Управление** | Централизованное (один оркестратор) | Децентрализованное (каждый сервис сам знает, что делать) |
| **Сложность** | Проще для старта, логика в одном месте | Сложнее отлаживать, логика распределена |
| **Масштабирование** | Оркестратор может стать узким местом | Лучше масштабируется, нет единой точки отказа |
| **Пример** | Один сервис-оркестратор вызывает другие сервисы по очереди | Сервисы публикуют события и реагируют на события других |

**Вывод:** Оркестрация проще для старта, хореография лучше масштабируется.

## 3. Что такое "компенсирующая транзакция"?

Компенсирующая транзакция — это действие, которое отменяет эффект предыдущей выполненной операции в саге. Она выполняется, когда последующий шаг саги завершается ошибкой.

**Пример:** Если шаг "Списание денег" не удался, компенсирующая транзакция для шага "Резервирование товара" — это возврат товара на склад (`releaseProduct`).

## 4. Может ли компенсирующая транзакция завершиться ошибкой?

Да, может. В таком случае применяются стратегии:

1. **Retry (повторные попытки)** — система автоматически повторяет компенсацию несколько раз (в вашем коде — 3 попытки с задержкой 1 секунда)
2. **Логирование ошибки** — запись в лог для дальнейшего анализа
3. **Dead Letter Queue** — помещение в очередь недоставленных сообщений
4. **Ручное вмешательство** — уведомление администратора


## 5. Что такое "согласованность в конечном счете" (Eventual Consistency)?

Eventual Consistency — это модель согласованности, при которой система не гарантирует мгновенную синхронизацию данных во всех сервисах, но гарантирует, что при отсутствии новых изменений все копии данных в конечном итоге придут к согласованному состоянию.

**Ключевые моменты:**
- Данные могут быть временно несогласованы
- Система продолжает работать, даже если не все сервисы синхронизированы
- Через некоторое время все сервисы увидят одинаковые данные

**Пример из саги:** После успешной оплаты заказ получает статус PAID, но информация об этом может дойти до сервиса уведомлений с задержкой. Через некоторое время клиент все равно получит уведомление.
